The Week He Brought MLG Fortress Back

April 25–14, 2026 (newest first)
An 8-year-old Minecraft server sat untouched until this month. Here's what happened when he decided to drag it into the modern era, usually with help from an AI.
Saturday, April 25 (Morning)

The Server Actually Feels Alive Again

He sat there Saturday morning looking at what got done this week. Forty-five plugin repos touched. Hundreds of commits. But the number that mattered was simpler: the server runs.

The custom stuff works. The minigames are playable. The shop doesn't lie about item names anymore. He'd taken an 8-year-old Minecraft server that could have just died quietly and got it running on modern software, keeping all its weird custom features, and actually working.

That's when the resurrection part was basically done.

Observation: He went from "this server is a museum piece" to "people can actually play on this" in about ten days. That was the win that mattered.
Friday, April 24

Multiverse-Core Reminded Him That Build Tools Change Too

He opened Multiverse-Core — the plugin that handles multiple worlds on the server. It's a big one, and he discovered it used to build with Maven but the project had switched to Gradle.

He asked an AI to completely rework the updater workflow. It wasn't just updating a plugin anymore — sometimes you have to update the way the plugin gets built. He spent the afternoon untangling that.

Another reminder that this isn't just "update plugin, done." It's more like "update plugin, update build system, update CI, hope nothing breaks, fix what breaks."

Observation: He handled the build tool switch without getting frustrated. Just another thing to fix, another layer to understand.
Thursday, April 23

He Found Some Weird Files and Figured Out What They Were

There were these strange image files in the server folder — world_biome.png, world_temperature.png, stuff like that. For a second he wondered if maybe they were part of some lost art project or custom terrain thing.

He asked an AI to investigate. Turns out they were just generated by Dynmap, the plugin that makes those web maps of your world. Totally disposable. He stopped treating them like precious artifacts and just recognized them for what they were: auto-generated clutter.

Reality check: Old server folders are full of mysteries. Most of them are just generated files from plugins you forgot you had.
Wednesday, April 22

The Shop Got One Last Fix, and He Simplified the Server Ping

He had an AI fix PrettySimpleShop one more time — it still wasn't using the actual shop item name in every case. Then he made a bigger change to MountainDewritoes: it used to rely on something called ServerListPlus to handle the server ping.

He ripped ServerListPlus out and made MountainDewritoes handle the ping directly. Fewer layers, less mystery, more control. There's a good feeling that comes from deleting a plugin instead of adding one.

Observation: He's at the point now where he's removing dependencies instead of adding them. That's a good sign the architecture is stabilizing.
Tuesday, April 21

CreativeParkour and the Rest Started Actually Working

He took on CreativeParkour — one of those classic MLG Fortress minigames. The plugin was so old it barely compiled. He had an AI remove old build workflows, clean up the Maven setup, and get it working with the modern server API.

By evening, he found and fixed a small crash bug in MountainDewritoes. The shift was noticeable: the big compatibility gaps were closed, so now he was finding the little bugs that only show up when players actually try to do things.

For players: CreativeParkour is that parkour minigame you probably played at some point. He fixed it so it won't break when the server updates.
Monday, April 20 (Evening)

He Had to Choose Between "Clean" and "Works" Again

Remember the player data changes from April 14? He had to undo part of it. The "clean" design he thought was so smart was making the main plugin file huge and hard to work with. So he brought back the wrapper approach.

He also fixed startup compatibility and trimmed down some config stuff. The plugin needed to load cleanly, and he knew all the fancy architecture in the world doesn't matter if the server crashes on startup.

This is the kind of thing that happens a lot in this work. You try something, it's elegant, then reality hits and you make it ugly but functional.

Monday, April 20 (Morning)

ItemNameLab Became Actually Useful

He had an AI create a test plugin called ItemNameLab. Sounds boring, right? But the logic was sound: with so many plugins needing to know item names (shops, GUIs, displays, etc.), he needed a way to test if the name logic worked everywhere.

He added tests for custom names, translated names, all that. The point wasn't to build a fancy tool. The point was to make sure that when he fixed item names in one plugin, they'd stay fixed in all of them.

Observation: He built a testing tool so he wouldn't have to manually check every plugin when he changed the name logic. Smart.
Sunday, April 19

PrettySimpleShop Could Finally Tell What Items Were

There was this bug: PrettySimpleShop (the shop plugin where players buy and sell stuff) couldn't correctly identify what an item was called. Like, you'd try to sell a diamond sword and the shop would call it something weird or wrong.

He spent the afternoon having an AI fix the item name fallback chain. Basically, the shop now knows: "Is this a custom named item? Okay, use that name. No? Is it a translated name? Use that. Still no? Then use the default Minecraft name." Simple in concept, annoying in practice.

When it was finally working, he logged into the test server and bought something just to make sure the shop signs showed the right names. They did. That felt good.

PrettySimpleShop item names fixed tested it himself
Thursday-Saturday, April 17-18

Everything Started Moving at Once

This was the turning point. Instead of fixing one plugin at a time, he started powering through the whole collection. There were like 45 plugin repos that needed attention — some hadn't been touched in years.

The routine became: open a plugin, update it to work with modern Minecraft (Paper/Purpur 26.1.2), fix the build system, remove old workflows, add new ones. Repeat. Over and over. Forty-five times.

By Saturday evening, the whole plugin org was starting to feel alive again. Stuff was compiling. More importantly, stuff was actually working on the modern server software.

Observation: He got into a rhythm. Open, update, fix, commit, next. The momentum was building and the server was coming back to life.
45 plugins build fixes modern Minecraft
Tuesday, April 15

The Build System Fought Back

He spent the morning fighting with build tools. It was like when you try to update something on your computer and suddenly nothing works. The plugins wouldn't compile because the Java version was wrong, then Maven (the build tool) was being loud and cranky.

He eventually got Java 21 working properly and fixed a bunch of repository URLs that pointed to nowhere. He also had to switch the updater script back on — the thing that checks for new versions.

He spent an entire day just fixing plumbing and had nothing to show for it in-game. But he understood: if it's not done, none of the fun stuff works.

The reality: Players never see this work. But if it's not done, none of the fun stuff works. It's like fixing the pipes in a house — boring but necessary.
Monday, April 14

GrandioseAPI Got Replaced With Something Simpler

He opened up the server's custom plugin folder and honestly, it was overwhelming. MountainDewritoes — yeah, that's really what he called it — had grown into this massive beast over the years. It handled player data, server tricks, and a bunch of stuff that made MLG Fortress feel like MLG Fortress.

MountainDewritoes depended on GrandioseAPI, a custom plugin he'd built to handle player data storage. It worked fine — stored everything in a `grandPlayers/` folder with UUID-named files. But Minecraft now has something called the PDC (Persistent Data Container) API built right in.

So he had an AI help him remove the GrandioseAPI dependency and switch to PDC directly in MountainDewritoes. Less plugins to maintain, same result. But then he realized the main plugin file was getting huge with the PDC code, so he brought back a small wrapper to keep things readable.

It was one of those days where every fix created a new problem. Classic.

For players: This plugin is why your player data doesn't vanish when you update the server. It's the backbone of a lot of the custom stuff you see in-game. He just simplified how it stores your data behind the scenes.
