The Week We Brought MLG Fortress Back

April 14–25, 2026
A diary of the resurrection. No technical jargon, just the story of how an 8-year-old Minecraft server got pulled into the modern era without losing its soul.
Monday, April 14

MountainDewritoes Was a Mess

I opened up the server's custom plugin folder and honestly, it was overwhelming. MountainDewritoes — yeah, that's really what we called it — had grown into this massive beast over the years. It handled player data, server tricks, and a bunch of stuff that made MLG Fortress feel like MLG Fortress.

The first problem? It depended on something called GrandioseAPI, which sounded fancy but was really just making everything more complicated. So I ripped it out. Player data got moved to something simpler (PDC, if you care about the technical side). But then I realized the main plugin file was getting huge, so I had to bring back a small wrapper to keep things readable.

It was one of those days where every fix created a new problem. Classic.

For players: This plugin is why your player data doesn't vanish when we update the server. It's the backbone of a lot of the custom stuff you see in-game.
Tuesday, April 15

The Build System Fought Back

Spent the morning fighting with build tools. You know how when you try to update something on your computer and suddenly nothing works? That was the vibe. The plugins wouldn't compile because the Java version was wrong, then the build tool (Maven, for those keeping score) was being loud and cranky.

Eventually got Java 21 working properly and had to fix a bunch of repository URLs that pointed to nowhere. The updater script — the thing that checks for new versions — also needed to be switched back on. It's funny how you can spend an entire day just fixing the plumbing and have nothing to show for it in-game.

Reality check: Players never see this work. But if it's not done, none of the fun stuff works. It's like fixing the pipes in a house — boring but necessary.
Thursday-Saturday, April 17-18

Everything Started Moving at Once

This was the turning point. Instead of fixing one plugin at a time, I started powering through the whole collection. There were like 45 plugin repos that needed attention — some hadn't been touched in years.

The routine became: open a plugin, update it to work with modern Minecraft (we're on Paper/Purpur 26.1.2 now), fix the build system, remove old workflows, add new ones. Repeat. Over and over. Forty-five times.

By Saturday evening, the whole plugin org was starting to feel alive again. Stuff was compiling. More importantly, stuff was actually working on the modern server software.

For players: This is when the server stopped feeling like a museum piece and started feeling like something that could actually run the latest Minecraft version.
45 plugins build fixes modern Minecraft
Sunday, April 19

PrettySimpleShop Could Finally Tell What Items Were

Here's a fun one. PrettySimpleShop is exactly what it sounds like — the shop plugin where players buy and sell stuff. But it had this annoying bug where it couldn't correctly identify what an item was called. Like, you'd try to sell a diamond sword and the shop would call it something weird or wrong.

Spent the afternoon fixing the item name fallback chain. Basically, the shop now knows: "Is this a custom named item? Okay, use that name. No? Is it a translated name? Use that. Still no? Then use the default Minecraft name." Simple in concept, annoying in practice.

When I finally got it working, I actually logged into the test server and bought something just to make sure the shop signs showed the right names. They did. Felt good.

PrettySimpleShop item names fixed actually tested it
Monday, April 20 (Morning)

ItemNameLab Became Actually Useful

Created a test plugin called ItemNameLab. Sounds boring, right? But here's the thing — with so many plugins needing to know item names (shops, GUIs, displays, etc.), I needed a way to test if the name logic worked everywhere.

Added tests for custom names, translated names, all that. The point wasn't to build a fancy tool. The point was to make sure that when I fixed item names in one plugin, they'd stay fixed in all of them.

Monday, April 20 (Evening)

MountainDewritoes Made Me Choose Between "Clean" and "Works"

Remember the player data changes from Monday last week? Yeah, I had to undo part of it. The "clean" design I thought was so smart was making the main plugin file huge and hard to work with. So I brought back the wrapper approach.

Also fixed startup compatibility and trimmed down some config stuff. The plugin needed to load cleanly, and all the fancy architecture in the world doesn't matter if the server crashes on startup.

This is the kind of thing that happens a lot in this work. You try something, it's elegant, then reality hits and you have to make it ugly but functional.

Tuesday, April 21

CreativeParkour and the Rest Started Actually Working

CreativeParkour is one of those classic MLG Fortress minigames. The plugin was so old it barely compiled. Spent the day removing old build workflows, cleaning up the Maven setup, and getting it to work with the modern server API.

By evening, I found and fixed a small NPE (null pointer exception, but basically just a crash bug) in MountainDewritoes. This is what the work looks like now — the big compatibility gaps are closed, so now I'm finding the little bugs that only show up when players actually try to do things.

For players: CreativeParkour is that parkour minigame you probably played at some point. It's still there, and now it won't break when we update the server.
Wednesday, April 22

The Shop Got One Last Fix, and the Server Ping Got Simpler

PrettySimpleShop needed one more correction — it still wasn't using the actual shop item name in every case. Fixed that. Then MountainDewritoes got another change: it used to rely on something called ServerListPlus to handle the server ping (that message you see when you look at the server in your multiplayer list).

I ripped ServerListPlus out and made MountainDewritoes handle the ping directly. Fewer layers, less mystery, more control. It's a good feeling when you can delete a plugin instead of adding one.

Thursday, April 23

Found Some Weird Files and Figured Out What They Actually Were

There were these strange image files in the server folder — world_biome.png, world_temperature.png, stuff like that. For a second I thought maybe they were part of some lost art project or custom terrain thing.

Nah. They were just generated by Dynmap, the plugin that makes those web maps of your world. Totally disposable. It felt good to stop treating them like precious artifacts and just recognize them for what they were: auto-generated clutter.

Lesson learned: Old server folders are full of mysteries. Most of them are just generated files from plugins you forgot you had.
Friday, April 24

Multiverse-Core Reminded Me That Build Tools Change Too

Multiverse-Core is the plugin that handles multiple worlds on the server. It's a big one — lots of players use it to have a survival world, a creative world, maybe a minigame world, all on the same server.

The problem? It used to build with Maven, but the project switched to Gradle. So the updater workflow — the thing that automatically builds new versions — needed to be completely reworked. It's not just updating a plugin anymore, sometimes you have to update the way the plugin gets built.

Spent the afternoon untangling that. Another reminder that this isn't just "update plugin, done." It's more like "update plugin, update build system, update CI, hope nothing breaks, fix what breaks."

Saturday, April 25 (Morning)

The Server Feels Alive Again

Sitting here on Saturday morning, looking at what got done this week. Forty-five plugin repos touched. Hundreds of commits. But the number that matters is: the server runs. The custom stuff works. The minigames are playable. The shop doesn't lie about item names anymore.

There's still cleanup to do in the main server folder — lots of config churn and generated file noise that needs to be sorted out. But the hard part, the resurrection part, that's basically done.

An 8-year-old Minecraft server that could have just died quietly is now running on modern software, keeping all its weird custom features, and actually working. That's a good feeling.

For players: You might not notice everything that changed, and that's the point. The server just works now. Your parkour is there, your shops work, your player data is safe. That's the win.
Written by someone who spent a week fighting with old plugins so you don't have to.
